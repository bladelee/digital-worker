## 二、Mattermost 连接器详解

### 2.1 什么是 Mattermost 连接器？

**Mattermost 连接器**是网关服务中负责与 Mattermost 服务器通信的组件。

### 2.2 使用的 API

| API | 用途 | 说明 |
|-----|------|------|
| **WebSocket API** | 接收实时消息 | `ws://mattermost:8065/api/v4/websocket` |
| **REST API** | 发送消息 | `POST /api/v4/posts` |
| **Bot API** | 管理数字员工 | `POST /api/v4/bots` |

### 2.3 与 Mattermost 的关系

```
┌─────────────────────────────────────────────────────────────────┐
│                      网关服务                                    │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │               Mattermost 连接器                          │   │
│  │                                                          │   │
│  │   WebSocket 连接 (接收消息)                              │   │
│  │   ├── 使用服务账号 token 认证                           │   │
│  │   ├── 监听 posted 事件 (私聊/提及)                       │   │
│  │   └── 监听 custom_* 事件 (群聊消息)                     │   │
│  │                                                          │   │
│  │   REST API (发送消息)                                    │   │
│  │   └── 使用 Bot token 发送消息到频道                     │   │
│  └──────────────────────────────────────────────────────────┘   │
│                              │                                   │
└──────────────────────────────┼───────────────────────────────────┘
                               │
            ┌──────────────────┼──────────────────┐
            │                  │                  │
     WebSocket            REST API           插件事件
     (实时接收)            (发送)            (广播)
            │                  │                  │
            ▼                  ▼                  ▼
┌─────────────────────────────────────────────────────────────────┐
│                    Mattermost 服务器                             │
│                                                                  │
│  ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐ │
│  │ WebSocket Hub   │  │ REST API Server │  │ 插件系统        │ │
│  └─────────────────┘  └─────────────────┘  └─────────────────┘ │
│                                                                  │
│  BotChannelForwarder 插件: 广播群聊消息到 custom_* 事件         │
└─────────────────────────────────────────────────────────────────┘
```

### 2.4 为什么需要插件？

| 场景 | 无插件 | 有插件 |
|-----|--------|--------|
| 私聊消息 | ✅ WebSocket 收到 `posted` | ✅ 同左 |
| @提及消息 | ✅ WebSocket 收到 `posted` | ✅ 同左 |
| **群聊普通消息** | ❌ 收不到 | ✅ 收到 `custom_*` 事件 |

**结论：** 插件是必须的，用于广播群聊消息。

## 三、成熟框架调研

### 3.1 WebSocket 连接池

| 框架 | 说明 | 适用性 |
|-----|------|--------|
| **websocket-pool** | 简单的 WebSocket 连接池，支持重连 | ⭐⭐⭐ 可用 |
| **ws** (自带) | Node.js 原生 WebSocket，需自己封装连接池 | ⭐⭐⭐⭐ 推荐 |
| **socket.io** | 功能丰富，但需要服务端配合 | ⭐⭐ 不适合 |
| **gorilla/websocket** (Go) | Go 语言最流行的 WebSocket 库 | ⭐⭐⭐⭐⭐ 如果用 Go |

### 3.2 消息路由框架

| 框架 | 说明 | 适用性 |
|-----|------|--------|
| **WAMP (autobahn)** | 标准化的 RPC 和 Pub/Sub 协议 | ⭐⭐⭐ 功能强大但复杂 |
| **Redis Pub/Sub** | 简单的消息发布/订阅 | ⭐⭐⭐⭐⭐ 推荐 |
| **NATS** | 高性能消息队列 | ⭐⭐⭐⭐⭐ 推荐 |
| **RabbitMQ** | 企业级消息队列 | ⭐⭐⭐ 功能丰富但重 |

### 3.3 推荐组合

**Node.js 方案:**
```
ws (WebSocket) + Redis (消息路由) + PostgreSQL (持久化)
```

**Go 方案:**
```
gorilla/websocket + NATS + PostgreSQL
```

## 四、网关服务详细设计

### 4.1 组件架构

```
┌─────────────────────────────────────────────────────────────────┐
│                      网关服务 (Gateway)                          │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  1. Mattermost 连接器                                     │   │
│  │     - WebSocket 客户端 (接收消息)                         │   │
│  │     - REST API 客户端 (发送消息)                          │   │
│  │     - 事件解析器 (posted, custom_*)                       │   │
│  └──────────────────────────────────────────────────────────┘   │
│                              │                                   │
│                              ▼                                   │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  2. 消息路由器                                            │   │
│  │     - 频道 → 数字员工映射                                 │   │
│  │     - 负载均衡 (Round-robin, Least-connections)          │   │
│  │     - 消息队列 (可选: Redis/NATS)                        │   │
│  └──────────────────────────────────────────────────────────┘   │
│                              │                                   │
│                              ▼                                   │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  3. OpenClaw 连接池                                       │   │
│  │     - 连接管理 (建立、心跳、重连、关闭)                   │   │
│  │     - 连接池 (按 worker_id 索引)                          │   │
│  │     - 消息发送/接收                                       │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  4. 租户管理 (简化版)                                     │   │
│  │     - 一个注册用户 = 一个租户                             │   │
│  │     - 配置存储 (PostgreSQL)                               │   │
│  └──────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

### 4.2 数据模型

```sql
-- 租户 (简化: 一个用户 = 一个租户)
CREATE TABLE tenants (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id VARCHAR(50) UNIQUE NOT NULL,  -- Mattermost user_id
    team_id VARCHAR(50),                   -- Mattermost team_id
    name VARCHAR(255),
    config JSONB DEFAULT '{}',
    created_at TIMESTAMP DEFAULT NOW()
);

-- 数字员工
CREATE TABLE workers (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id UUID REFERENCES tenants(id),
    name VARCHAR(255) NOT NULL,
    type VARCHAR(50) DEFAULT 'assistant',  -- assistant, developer, sales, etc.
    mattermost_bot_id VARCHAR(50),         -- Mattermost Bot ID
    openclaw_url VARCHAR(255),             -- OpenClaw WebSocket URL
    openclaw_token VARCHAR(255),           -- OpenClaw 认证 token
    status VARCHAR(20) DEFAULT 'active',
    created_at TIMESTAMP DEFAULT NOW()
);

-- 频道配置
CREATE TABLE channel_configs (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id UUID REFERENCES tenants(id),
    channel_id VARCHAR(50) NOT NULL,       -- Mattermost channel_id
    worker_ids UUID[] DEFAULT '{}',        -- 关联的数字员工
    config JSONB DEFAULT '{}',
    UNIQUE(tenant_id, channel_id)
);
```

### 4.3 消息流程

```
1. Mattermost 用户发消息
   │
   ▼
2. Mattermost 服务器
   ├── 私聊/提及 → posted 事件
   └── 群聊 → 插件广播 custom_* 事件
   │
   ▼
3. 网关 Mattermost 连接器
   ├── 接收 WebSocket 事件
   ├── 解析事件类型
   └── 提取消息内容
   │
   ▼
4. 消息路由器
   ├── 根据 channel_id 查找关联的 workers
   ├── 负载均衡选择实例
   └── 构建消息包
   │
   ▼
5. OpenClaw 连接池
   ├── 获取目标连接
   └── 发送消息到 OpenClaw
   │
   ▼
6. OpenClaw 处理
   ├── LLM 推理
   └── 返回响应
   │
   ▼
7. 网关接收响应
   ├── 通过 REST API 发送到 Mattermost
   └── 使用 Bot 身份发送消息
   │
   ▼
8. Mattermost 用户看到响应
```

## 五、网关服务替代方案

### 方案 D-1: 纯 Node.js (当前推荐)

```
Node.js + ws + Redis + PostgreSQL
```

**优点:**
- 开发快速
- 生态丰富
- 与 OpenClaw 技术栈一致

**缺点:**
- 单线程 (需要 cluster 模式)
- 性能不如 Go

---

### 方案 D-2: Go 语言实现

```
Go + gorilla/websocket + NATS + PostgreSQL
```

**架构:**
```
┌─────────────────────────────────────────────────────────────────┐
│                      Go Gateway                                 │
│                                                                  │
│  ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐ │
│  │ Mattermost      │  │ 消息路由器      │  │ OpenClaw        │ │
│  │ Connector       │  │ (NATS)          │  │ Connection Pool │ │
│  │ (gorilla/ws)    │  │                 │  │                 │ │
│  └─────────────────┘  └─────────────────┘  └─────────────────┘ │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │ 协程池 (ants) - 高并发处理                               │    │
│  └─────────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────────┘
```

**优点:**
- 高性能
- 原生并发
- 低内存占用
- 编译为单一二进制

**缺点:**
- 开发效率低于 Node.js
- 需要学习 Go

**适用场景:** 大规模部署 (1000+ 并发)

---

### 方案 D-3: 基于 Centrifugo

**Centrifugo** 是一个开源的实时消息服务器。

```
┌─────────────────────────────────────────────────────────────────┐
│                      Centrifugo (消息总线)                       │
│                                                                  │
│  ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐ │
│  │ Mattermost      │  │ OpenClaw        │  │ 管理后台        │ │
│  │ Connector       │  │ Connector       │  │                 │ │
│  │ (Publisher)     │  │ (Subscriber)    │  │                 │ │
│  └─────────────────┘  └─────────────────┘  └─────────────────┘ │
└─────────────────────────────────────────────────────────────────┘
```

**架构:**
```
Mattermost → Connector → Centrifugo → OpenClaw instances
                  ↓
            PostgreSQL (配置)
```

**优点:**
- Centrifugo 处理消息路由
- 支持水平扩展
- 有管理界面
- 支持多种协议 (WebSocket, SSE, HTTP-stream)

**缺点:**
- 引入额外依赖
- 学习成本

**适用场景:** 需要管理界面和多协议支持

---

### 方案 D-4: 基于 Temporal

**Temporal** 是一个工作流编排引擎。

```
┌─────────────────────────────────────────────────────────────────┐
│                      Temporal Server                             │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │ 工作流: 消息处理工作流                                   │    │
│  │                                                          │    │
│  │   接收消息 → 路由 → 调用 OpenClaw → 超时处理 → 发送响应 │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                  │
│  ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐ │
│  │ Mattermost      │  │ OpenClaw        │  │ 重试/补偿       │ │
│  │ Activity        │  │ Activity        │  │ Activity        │ │
│  └─────────────────┘  └─────────────────┘  └─────────────────┘ │
└─────────────────────────────────────────────────────────────────┘
```

**优点:**
- 自动重试
- 工作流可视化
- 支持长时间运行的任务
- 内置超时处理

**缺点:**
- 复杂度高
- 资源消耗大
- 学习曲线陡峭

**适用场景:** 复杂业务流程，需要可靠性和可观测性

## 七、方案 3 vs 方案 4 的最新选择

### 7.1 方案对比

| 维度 | 方案 3 (全新开发) | 方案 4 (混合网关) |
|-----|------------------|------------------|
| **开发周期** | 6个月+ | 6周 |
| **技术风险** | 高 | 低 |
| **功能完整性** | 最高 | 中等 |
| **维护成本** | 低 | 中等 |
| **依赖 Mattermost** | 否 | 是 |
| **用户体验** | 需要开发客户端 | 使用 Mattermost 客户端 |

### 7.2 最新建议

**推荐: 方案 4 (混合网关)**

**理由:**
1. **验证成功** - 插件 + 用户 token 方案已验证可行
2. **快速落地** - 6周可完成 MVP
3. **用户体验好** - 使用成熟的 Mattermost 客户端
4. **可演进** - 未来可以迁移到方案 3

**实施路径:**
```
阶段 1 (6周): 方案 4 MVP
    ↓ 验证业务
阶段 2 (3个月): 优化网关 (Go 重写)
    ↓ 规模化
阶段 3 (1年): 评估是否需要方案 3
```
